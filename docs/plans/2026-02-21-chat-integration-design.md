# Дизайн интеграции с Chat Service

**Дата:** 2026-02-21
**Статус:** Утвержден

## 1. Контекст

Текущему бэкенду `leaf-flow` необходимо интегрироваться с новым микросервисом чата для **отправки системных событий**.
Когда в системе `leaf-flow` происходят важные действия (например, `order.created`, `order.status_changed`), мы должны отправлять соответствующие события в `Redis Stream` (`leaf.events`). Микросервис чата вычитает их оттуда и создаст диалоги или системные сообщения.

*Примечание: Получение событий из чата (chat.fanout / Pub-Sub) в рамках данного репозитория не реализуется, это зона ответственности другого микросервиса.*

## 2. Архитектура интеграции (Отправка событий)

Для гарантии консистентности данных мы будем использовать существующий в проекте паттерн **Outbox**, точно так же, как сейчас отправляются уведомления при создании и изменении статуса заказа.

### Механика работы:
1. **Запись в `outbox_messages`:** В бизнес-логике (например, в сервисе заказов), в рамках одной транзакции с изменениями БД, создается запись в таблице `outbox_messages`. Это поведение уже реализовано для событий заказов.
2. **Обработка `OutboxProcessor`:** Существующий фоновый воркер `leaf-flow-outbox-worker` асинхронно читает необработанные события из outbox в БД.
3. **Обработчики событий:** В существующие обработчики (которые отвечают за пересылку событий в Celery) мы добавим код для отправки этих событий еще и в `Redis Stream` (`leaf.events`).
4. **Гарантии:** Так как `OutboxProcessor` отслеживает успешность обработки, мы гарантируем, что событие в чат будет отправлено ровно в том случае, если бизнес-операция была успешной (At-least-once delivery).

## 3. Решение и подход к реализации

1.  Создать утилиту `send_chat_event` или интеграционный класс `ChatEventPublisher`, который будет инкапсулировать вызов `aioredis.xadd("leaf.events", payload)`. Будем использовать текущий клиент Redis (доступен глобально или инстанцируется).
2.  Найти текущих подписчиков `OutboxProcessor`, которые реагируют на создание заказа или изменение статуса (см. `src/leaf_flow/services/notification/`).
3.  Обогатить эти обработчики вызовом `ChatEventPublisher`.
4.  Собрать нужный `payload` для чата согласно формату `docs/integration-backend.md`:
    *   **order.created**: `event_type`, `user_id`, `order_id`
    *   **order.status_changed**: `event_type`, `order_id`, `status`, `old_status`
